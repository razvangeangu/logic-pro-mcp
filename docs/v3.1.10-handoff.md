## What was actually broken (root cause depth)

### Level 1: AppleScript dictionary changes in Logic 12.x (the user-reported symptom)

```
$ osascript -e 'tell application "Logic Pro" to tell front document to return count of tracks'
→ -2753: tracks 변수를 정의하지 않았습니다.
```

Apple stripped `tracks`, `markers`, `tempo`, `time signature` from Logic Pro 12.x's AppleScript scripting dictionary. The v3.1.5 helpers that relied on these terms returned -2753 on every call. v3.1.5 / v3.1.6 / v3.1.7 closed Issues #3 / #4 / #5 without verifying on 12.x — the AX fallback (which had its own pre-existing defects) was the only path that ran.

### Level 2: AX surface drift between Logic 12.0/12.1 and 12.2

- **Inspector subtree leak (#3)**: `getTrackHeaders` had an outline/table fallback that matched any `AXOutline` when `Track Headers` identifier wasn't found. On Mixer-focused Logic 12.x, that fallback grabbed the Inspector outline — which contains AXGroup field rows like `Mute:`, `Loop:`, `Quantize:` — and `extractTrackName` surfaced those as track names.
- **Marker ruler missing (#5 on 12.0/12.1)**: AX path required the marker ruler element to carry an identifier/description containing `marker` / `마커`; 12.x removed that tag. v3.1.8's hardened walker tried `AXRuler` structural position as an alternative.
- **AXRuler removed entirely (#5 on 12.2)**: but Apple removed `AXRuler` from the arrange window's AX subtree on 12.2 (verified `osascript`: zero `AXRuler` elements). User markers only exist in the dedicated **Marker List** window's `AXTable`. v3.1.9 added that scrape.

### Level 3: cache-staleness bug (`StateCache.updateMarkers`)

While testing v3.1.9 with a closed marker list:
```swift
func updateMarkers(_ newMarkers: [MarkerState]) {
    guard markers != newMarkers else { return }  // ← bug
    markers = newMarkers
    markersFetchedAt = Date()
}
```

The equality short-circuit skipped the `markersFetchedAt` update for empty-to-empty polls. The cache's resource handler then reported `source: "default"` (no poll has run) instead of `source: "ax_live"` (poll ran, observed nothing) — making "honest empty" indistinguishable from "never polled". Fixed: timestamp always advances; data assignment stays equality-guarded.

### Level 4: dispatcher routing bug (`NavigateDispatcher.goto_marker`, boomer-caught P1)

Final BOOMER-6 review on v3.1.9:

```swift
if let marker = markers.first(where: { $0.name.localizedCaseInsensitiveContains(name) }) {
    let result = await router.route(
        operation: "nav.goto_marker",
        params: ["index": String(marker.id)]
    )
    return toolTextResult(result)
}
```

The dispatcher correctly resolved the marker from cache, then routed `nav.goto_marker { index: <id> }` to `MIDIKeyCommandsChannel`. But that channel ignores all params and fires fixed CC 38 — Logic's "go to next marker" hotkey. So both index- and name-based goto silently advanced the marker pointer by one regardless of which marker the cache matched. v3.1.10 routes via `transport.goto_position { position: marker.position }` instead.

This bug was latent in v3.1.0–v3.1.8 because the cache was always empty (the broken AX walker never populated it). v3.1.9's marker list scrape made the cache real, surfacing the dispatcher bug.

## Key architectural decisions

### Tier-merge at the resource handler, not the channel (v3.1.8 boomer P0)

Original draft of v3.1.8 had the AX channel synthesise placeholder track rows when AX returned empty. boomer review: this would let `StatePoller` write placeholders into shared `StateCache`, which name-routed write actions (`track.select { name }`) consult. Caller types `name: "Track 5"` → matches placeholder → routes write to a fake track index. Data integrity hazard.

**Fix**: tier-merge moved to `ResourceHandlers.readTracks` / `readProjectInfo`. Cache stays placeholder-free; resource layer synthesises rows only into the response without touching cache. Resource reads are read-only with respect to cache; the poller is the sole writer.

### Marker walker strategy ordering (v3.1.9)

```
1. Marker List window AXTable          ← Logic 12.2+ (only place markers live)
2. Arrange-area AXRuler structural     ← Logic 11.x (preserved compat)
3. AXGroup keyword match               ← Logic 10.x (legacy)
```

Strategy 1 is preferred even when other strategies could match — minimises the chance of stale ruler data overriding the canonical list.

### `goto_marker` cold-cache fallback (v3.1.10)

When the cache is empty (poller hasn't run yet, or marker list window closed), index-based callers fall through to the legacy `nav.goto_marker` keycmd path so they get *some* navigation. Name-based callers with cold cache return a clear `No marker found matching '<name>'` error because the keycmd has no name-aware semantic.

### 13-locale title-suffix matrix (v3.1.9 guardian P2)

Initial draft had `markerListWindowSuffixes = ["- 마커 목록", "- Marker List"]`. Guardian flagged: Logic Pro is localised to ~12 languages. Extended to 13 locales (KR/EN/JA-x2/FR/DE/ES/IT/ZH-S/ZH-T/RU/PT/NL). Same expansion applied to `markerCellPlaceholders` set (the localised word for "cell" — placeholder strings AXCell carries by default).

## Known limitations (deferred)

Per `PRD-issue7-logic12-read-paths.md` Non-Goals:

- **NG1**: Track names from project file. `Alternatives/000/ProjectData` is a binary blob (custom format header `#G\xc0\xab\xd0\x09`). Reverse-engineering deferred. Placeholder rows path is the documented honest fallback.
- **NG2**: Marker positions / names from project file. Same as NG1.
- **NG8**: Per-section document-identity contract (cross-project cache contamination on rapid switches). Existing cache invalidation on `hasDocument: false` is sufficient for v3.1.x.
- **NG9**: Tri-state marker result (`unreadable` vs `none`). `ax_occluded: true` envelope flag is the existing untrusted-empty signal.
- **Auto-open Marker List window**: planned `LOGIC_PRO_MCP_AUTO_OPEN_MARKER_LIST=1` env-gated behavior; not shipped — file an issue if you want it.

## Process learnings

The v3.1.5 → v3.1.7 chain shipped three releases that all claimed to fix Issues #3 / #4 / #5 without ever verifying against a real Logic 12.x install. The reporter did the verification each time and reopened. The break-the-loop pattern was:

1. **Reproduce on the actual reported version** (Logic 12.0.1 + 12.2 both behaved identically for the dictionary-emptiness issue)
2. **Read the upstream evidence** (the reporter's `sdef` dump in #7 showed exactly which terms were missing)
3. **End-to-end test the new binary**, not just unit tests (catches the dispatcher routing bug at level 4 — boomer iteration 7 spotted what 6 prior reviews missed)
4. **Honestly disclose what wasn't tested** (v3.1.9 release notes explicitly said "12.2 verification deferred to user" because owner was on 12.0.1 — the reporter then verified end-to-end)

Total review iterations across all releases: **8 BOOMER-6 + 4 guardian + 4 tester** passes. The final boomer iteration on v3.1.9 caught the goto_marker silent no-op that 7 prior reviews + the v3.1.9 ship had missed. The cycle was worth it.
